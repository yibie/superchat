# 本地记忆层的一次个人探索：在有限条件下实现 memory layer

**摘要：** 这篇文章记录的不是一个已经成熟的长程问答系统，而是我对本地记忆层的一次个人探索。我关心的问题很具体：在完全本地、资源有限、强调可审计与可编辑的条件下，能不能先把 memory layer 这一层做出来。围绕这个问题，我尝试用 Org 文本承载记忆，用 `org-ql` 做候选召回，用显式关系和弱关键词关联补足跨条目连接，再用已有测试去看这条路到底能走到哪里。结果并不意味着问题已经被解决，但至少说明了一件事：本地记忆层不是空想，它可以成立；同时，它的弱势也很清楚，尤其一旦进入真实长程对话场景，时间建模、状态跟踪和排序能力会迅速成为瓶颈。

**关键词：** 本地记忆层、Org-mode、org-ql、Superchat-Memory、Emacs

---

## 一、我为什么想做本地记忆层

我对这个方向的兴趣，不是因为我想重新发明一个完整的 long-horizon QA system，而是因为我始终觉得，记忆层和问答层不应该被混成一件事。

很多现有方案默认从向量数据库、云端 API、embedding pipeline 开始。这类方案在语义召回上当然有优势，但它们也天然带来几个我不太愿意接受的前提：数据要离开本机，检索过程比较黑盒，系统一旦依赖外部服务，长期可维护性也会被外部条件牵着走。

我想探索的方向更朴素一些：如果约束条件非常严格，比如必须完全本地运行、必须能直接查看和修改原始记忆、必须尽量减少外部依赖，那么能不能先做出一个够用的 memory layer？

这里的关键词是 memory layer，而不是 QA system。前者负责的是记忆如何落地、如何检索、如何组织；后者则还要解决时间推理、状态更新、多证据综合和最终回答生成。我的兴趣更偏向前一层，因为我想先确认，在有限条件下，记忆本身能否先成立。

---

## 二、我的约束条件是什么

这次探索从一开始就不是在宽松环境里做的。它的前提条件很明确：

1. 不依赖云端 API 作为系统成立的必要条件。
2. 优先使用本机 CPU 和本地模型环境。
3. 记忆必须可审计，也就是我能直接看到它到底存了什么。
4. 记忆必须可编辑，也就是系统犯错时，我可以手动修正。
5. 结构要尽量简单，不能为了“看起来更智能”而把系统复杂度推得过高。

在这样的前提下，很多常见路线其实一开始就被我排除了。比如，完全依赖 embedding 的方案不适合作为核心前提；过重的数据库设计也不符合我想要的“人能直接接管”的目标。

所以最后选出来的路线，是把记忆落到 Org 文本里，用规则与结构化属性去支撑检索和扩展。这条路线听起来保守，但它有一个我很看重的优点：它把“机器能处理”和“人能理解”放在了同一个载体上。

---

## 三、为什么我选择 Org-mode，而不是普通纯文本

如果只是把记忆写进一个文本文件，那还不够。我要的不是一堆无结构的段落，而是既能直接阅读，又能被程序稳定处理的结构化记忆。

这正是 Org-mode 对我有吸引力的地方。它表面上是文本，实际上却内含清晰的结构层次：headline、property drawer、timestamp、tags、正文内容，都有稳定语义。更重要的是，Emacs 可以把这些内容解析成 AST。也就是说，这不是“看起来像文本的数据文件”，而是一种天然可编程的结构化文档。

在这个意义上，Org-mode 的价值不是“写起来方便”这么简单，而是它同时满足了两件事：

1. 人可以直接打开看，知道系统到底记住了什么。
2. 程序可以稳定地把它解析成结构化节点，再做检索、排序和关系扩展。

我在这次实现里采用的记忆条目，大体就是下面这种形态：

```org
* Alice uses VS Code :PERSON:TOOL:
:PROPERTIES:
:ID:         abc123
:TIMESTAMP:  [2026-03-19 20:10:00]
:TYPE:       insight
:KEYWORDS:   alice, vscode, developer, seattle
:RELATED:    def456, ghi789
:END:
Alice mentioned that she uses VS Code for daily development work in Seattle.
```

这类结构的好处很直接。对人来说，它已经足够可读；对程序来说，它又天然包含了 title、tags、timestamp、keywords、relations 这些可以直接利用的字段。换句话说，Org 条目本身就兼具“记忆文本”和“轻量数据库记录”的双重属性。

这也是为什么我觉得 Org-mode 不只是一个存储选择，而是这条本地 memory layer 路线能不能成立的基础之一。

---

## 四、我的设计取舍：不是追求最强，而是先让它成立

在实现层面，我做的不是一个“大而全”的系统，而是一系列很明确的取舍。

### 4.1 分层写入，而不是假装系统总能自动理解一切

我把记忆写入分成几层，本质上是在承认一件事：系统对“什么值得记住”的判断并不可靠，所以不能把所有希望都压在自动提取上。

显式指令是最强信号。用户明确说“记住”，就意味着这条信息在主观上已经被标记为重要，这时系统应当优先尊重这个判断。启发式自动捕获可以存在，但它只能是补充。至于回顾性整理和手工修正，我认为更不是缺点，而是本地记忆系统必须保留的能力。

如果一个 memory layer 连用户手工修正都不欢迎，那它更像是把记忆从用户手里夺走，而不是在帮助用户管理记忆。

### 4.2 关键词、时间衰减与显式关系，优先于复杂语义管线

我没有把语义检索当作这个实验的起点。相反，我先使用关键词、标签、时间戳和访问计数这些确定性更强的东西来组织检索。

这样做的直接好处是简单、稳定、可解释。相同输入会产生相同关键词；排序逻辑可以直接看；记忆为什么被召回，也更容易追溯。代价当然也很明显：这种方式在语义泛化和复杂推理上明显更弱。

但在我看来，作为一个本地 memory layer 的第一步，这种取舍是合理的。因为我想先解决“记忆能不能被存得清楚、取得出来、关系能不能串起来”这些更基础的问题，而不是一开始就把整个系统做成一个高度复杂的黑盒。

### 4.3 显式 `:RELATED:` 比自动关系发现更符合这次探索的目标

如果目标是完全自动化，那显式关系显然不够“聪明”。但如果目标是本地、可控、可审计，那显式关系其实很合适。

我引入 `:RELATED:` 的原因不是因为它最先进，而是因为它把关系的解释权留在了人这边。两条记忆为什么相连，不需要依赖模型的隐式判断，而是能在文本上直接看到。

这会牺牲一些自动化便利，但换来的是维护成本可控、行为边界清楚。对这次探索来说，这个交换是值得的。

### 4.4 关键词关联发现只是弱补充，不是主角

我后来又加了关键词关联发现，是因为单靠 direct retrieval 和显式关系，还不足以覆盖所有实际场景。有些相关事实没有显式 `:RELATED:`，但确实共享了一些稳定关键词，这时候弱关联补充是有意义的。

不过我很清楚，这个机制最多只能算补丁，不能当主干。它能补 recall，但不能替代状态判断、时间推理和更可靠的 reranking。一旦把它抬到主角位置，噪声就会迅速上升。

## 五、从代码看，superchat-memory 实际上在做什么

如果只看名字，`superchat-memory` 容易被误解成一个已经具备复杂问答能力的“记忆系统”。但从实现主路径看，它更像是一个检索与组织层。

最能说明问题的是 `superchat-memory-retrieve` 这段代码：

```elisp
(let* ((terms (superchat-memory--prepare-search-terms query-string))
       (direct-entries (if (featurep 'org-ql)
                           (superchat-memory--retrieve-with-org-ql terms)
                         (superchat-memory--retrieve-fallback terms)))
       (expanded (superchat-memory--expand-with-related direct-entries))
       (associated (when superchat-memory-relation-discovery-enabled
                     (superchat-memory--discover-keyword-associations
                      direct-entries terms)))
       (all-entries (cl-remove-duplicates
                     (append expanded associated)
                     :key (lambda (e) (plist-get e :id))
                     :test #'string-equal))
       (ranked (superchat-memory--rank-entries all-entries terms)))
  ranked)
```

这段流程非常关键，因为它准确暴露了这个系统目前处在哪一层：

1. 先把 query 变成 search terms。
2. 用这些 terms 做 direct retrieval。
3. 沿显式 `:RELATED:` 扩展到相邻记忆。
4. 如果启用了关键词关联，再做一次弱补充召回。
5. 最后统一排序，返回一组候选条目。

这里没有出现真正意义上的 long-horizon answer synthesis。系统做的是“给出更可能相关的记忆”，而不是“完成复杂问题的最终推理和回答”。这也是我为什么坚持把它叫作 memory layer，而不是 long-horizon QA system。

写入路径也体现了同样的设计方向。记忆最终不是进入某种不可见的内部索引，而是直接落成 Org headline 与 property drawer：

```elisp
(insert (format "* %s%s\n" (superchat-memory--truncate-title title) tag-str))
(insert ":PROPERTIES:\n")
(insert (format ":ID:       %s\n" entry-id))
(insert (format ":TIMESTAMP: %s\n" ts))
(insert (format ":ACCESS_COUNT: %d\n" (max 0 final-access)))
(when final-keywords
  (insert (format ":KEYWORDS: %s\n" (mapconcat #'identity final-keywords ", "))))
(when final-related
  (insert (format ":RELATED:  %s\n" (mapconcat #'identity final-related ", "))))
(insert ":END:\n")
```

这段代码背后其实是一种很明确的想法：记忆不是只给机器用的内部状态，而是应该成为一种人和系统都能共同操作的中间层。也正因为如此，这个实现的价值不只是“检索快不快”，还在于它让记忆本身保持了可读、可查、可修订。

## 六、org-ql 在这里的作用，不是推理，而是候选召回

如果说 Org-mode 解决的是“记忆怎样被存成结构化文本”，那 `org-ql` 解决的就是“这些结构化条目怎样被有效找出来”。

我在这次实现里依赖 `org-ql`，不是因为它能自动推理，而是因为它能把 Org 文档里的结构直接转成可执行查询。没有它，系统就只能退回到对全文的线性扫描；有了它，检索可以围绕 headline、tags、regexp 和属性组织。

对应的核心代码如下：

```elisp
(defun superchat-memory--retrieve-with-org-ql (query-or-terms)
  (let ((terms (superchat-memory--ensure-terms query-or-terms)))
    (when terms
      (let ((query
             `(or ,@(mapcar
                     (lambda (term)
                       (let* ((regex (plist-get term :regex))
                              (tag (plist-get term :tag))
                              (clauses (delq nil
                                             (list (and regex (list 'regexp regex))
                                                   (and tag (list 'tags tag))))))
                         (cons 'or clauses)))
                     terms))))
        (org-ql-select buffer query
                       :action #'superchat-memory--org-ql-entry)))))
```

这个过程的本质很简单：系统先把 query 变成 terms，再把每个 term 组织成 `regexp` 和 `tags` 查询子句，最后交给 `org-ql-select` 去找候选条目。找到之后，再统一转成 memory plist，供后面的排序和扩展继续处理。

所以在我的理解里，`org-ql` 在这里承担的是候选召回层，而不是理解层。它让 memory layer 从“能存”走向“能查”，但它并不替系统解决时间推理、状态冲突和最终答案综合。也正是因为这一点，后面的实验结果才会显得很有代表性：候选召回可以做得不错，但一进入更真实的长程场景，单有召回还远远不够。

## 七、实验对我说明了什么

我这次保留测试，不是为了把文章写成 benchmark 汇总，而是因为没有测试，很难判断这条本地 memory layer 路线到底是在自我感觉良好，还是确实有一点可行性。

### 7.1 EverMemBench 说明了“它可以成立”

EverMemBench 这部分对我最有价值的地方，在于它验证了一个相对积极的事实：在合成多跳关系场景里，本地 memory layer 确实可以做出可用效果。

尤其是引入显式关系扩展之后，Multi-hop 通过率从 `20%` 提升到 `100%`，平均检索时间从约 `8.5 ms` 增加到约 `12 ms`。这个结果并不意味着系统变成了强推理系统，但它至少证明了：只要记忆结构被组织得足够清楚，显式图式扩展在本地环境中是能工作的。

对我来说，这个结论很重要，因为它说明本地 memory layer 并不是只能做单条 recall。只要结构设计得当，它确实能够在一定范围内支持跨条目的弱多跳能力。

### 7.2 关键词关联测试说明了“弱补充是有价值的，但价值有限”

在专门构造的关键词关联测试里，关闭关键词关联发现时，`11` 个 QA 通过 `10` 个；开启后，`11` 个 QA 全部通过，且 `via-assoc = 1`。这说明关键词关联不是完全无用的花哨开关，它在某些桥接场景里确实能补回遗漏信息。

但这个结果同样提醒我，它更适合被理解成弱补充机制。它可以在 direct hit 已经成立的前提下，把附近一些潜在相关条目带回来；但如果问题本身要求的是时间推理或状态判断，那么单靠关键词共现并不能解决核心困难。

### 7.3 LoCoMo 切片说明了“它的弱势在哪里”

真正让我看到边界的，是 LoCoMo 切片。

在固定 200 个 QA 的切片上，关闭关键词关联时，`answer-hit = 35.0%`，`evidence-hit = 39.0%`，平均检索约 `1786.79 ms/query`；开启关键词关联后，`answer-hit = 33.5%`，`evidence-hit = 39.0%`，平均检索约 `2527.09 ms/query`，并且 `via-assoc = 90`。

这个结果我认为非常诚实。它说明关键词关联确实触发了很多次，但这些触发并没有转化成更好的证据命中，反而拖慢了检索，并让 answer hit 略微下降。

这意味着问题的核心已经不再是“有没有找到更多相关条目”，而是：

1. 哪条信息对应当前状态。
2. 哪条信息只是过去状态。
3. 多个 session 中分散的事实，哪部分才是回答问题真正需要的证据。
4. 候选记忆找到之后，系统如何判断它们的重要性。

从这个角度看，LoCoMo 对我的意义不是告诉我“这个系统分数高不高”，而是告诉我：本地 memory layer 可以成立，但它一旦面对真实长程对话，弱势会非常集中地暴露在时间线恢复、状态跟踪和排序上。

---

## 八、这篇探索真正想说明的，不是成绩，而是方向

如果让我概括这次探索最核心的收获，那不是“我做出了一个能和线上系统竞争的方案”，而是更克制的一点：在有限条件下，本地 memory layer 是可以被做出来的，而且它有实际意义。

它的意义首先在于可控。记忆并不被封装在外部服务里，而是落在我可以直接查看和编辑的结构里。其次在于可审计。系统为什么召回一条记忆，通常可以从关键词、标签、关系或排序逻辑里追溯出来。最后在于可维护。即使自动化部分不完美，人仍然可以接管它，而不是被它绑架。

但我也不认为这就足够了。`superchat-memory` 当前展示出来的，更接近一个成立了的 memory layer，而不是一个完整的 long-horizon QA system。它可以把记忆组织起来，可以做一定程度的候选召回和弱多跳扩展，但它还不能稳定地处理真实长程对话里的状态演化、时间判断和最终证据综合。

所以这篇文章真正想表达的，不是“问题解决了”，而是“这一层值得先被单独做出来”。只有当这层成立之后，后面再叠加 reranking、状态建模、本地大模型约束推理，才会有更扎实的基础。

---

## 九、结语

这次探索让我更确信一件事：在记忆系统里，把 memory layer 和问答层分开看，是有必要的。

前者关心的是，记忆能否以本地、可审计、可编辑的方式存在；后者关心的是，系统能否在长时间跨度问题上做出高质量回答。这两层当然相关，但它们不是一回事。

我现在做出来的 `superchat-memory`，更适合被理解为前一层的一次实践验证。它证明了，在 Org-mode、`org-ql`、显式关系和有限规则检索的组合下，本地记忆层是有可能成立的。它也同样提醒我，这条路线的下一步，不应该只是继续堆关键词技巧，而应该认真面对时间建模、状态跟踪和 reranking 这些真正决定上限的问题。

如果把这篇文章当作一次阶段性记录，我希望它留下的不是“我做成了什么终局方案”，而是一个更值得继续讨论的问题：在不放弃本地性、可控性和可审计性的前提下，memory layer 最合理的形态到底应该是什么。
