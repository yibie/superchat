# PR/FAQ Draft: Resume Recovery

## Press Release

### Headline
Threadnote turns Resume into a restart point, not a summary wall.

### Subtitle
When a user reopens a thread, Threadnote now restores the smallest useful state needed to continue work in seconds.

### Date
March 11, 2026

### Intro paragraph
Today we are starting the next Threadnote phase: Resume Recovery. This phase focuses on the product's real core job. Instead of showing a structurally correct but cognitively heavy resume surface, Threadnote will collapse recovery into a minimal restart point that helps a user re-enter a thread in 10-30 seconds.

### Problem paragraph
Most note products fail at continuity. They preserve information but do not preserve working state. Even after adding goal-aware Resume components, a user can still reopen a thread and face too much visible structure. The result is a reading task, not a restart point.

### Solution paragraph
Resume Recovery fixes this by making Resume aggressively small by default. Each thread will open with only three goal-shaped lines: the current state, the most important missing piece, and the next best move. Everything else moves behind a controlled second layer anchored on resolved questions and settled decisions.

### How the product works
Every thread still carries a single goal layer. But the default Resume no longer expands into a component wall. Instead, Threadnote selects three plastic recovery lines based on the thread's goal type. Users can then open a narrow “Resolved So Far” layer to confirm what has already been decided or ruled out, without falling back into full archive reading.

## FAQ

### Internal FAQs

**Q: Why is this a new phase instead of finishing the current plastic resume phase first?**  
A: Because this is a product-level correction, not a visual tweak. The previous phase established goal-aware infrastructure. This phase changes the actual recovery model and the user's information boundary.

**Q: What is the core objective of this phase?**  
A: Make Resume behave like a restart mechanism: one screen, three lines, one controlled expansion boundary.

**Q: What is intentionally out of scope?**  
A: Full stream redesign, resource-system redesign, broad AI improvements, and generic dashboard features.

**Q: What does success look like?**  
A: A user reopens a thread, reads three lines, and can continue immediately without scanning a wall of context.

### Customer FAQs

**Q: Will I lose context if Resume only shows three lines?**  
A: No. The product will still keep supporting detail, but it will no longer force all of it into the default Resume.

**Q: What happens to everything that is already solved?**  
A: It moves into a secondary “Resolved So Far” layer so you can confirm what is settled without rereading the thread.

**Q: Does this make Resume more AI-driven?**  
A: AI still helps shape recovery, but it stays constrained. The product decides the boundary: three default lines, then one controlled expansion path.
